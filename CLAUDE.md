## Project Overview

Navtrack is an open-source GPS tracking system with a .NET 10 backend and React frontend. It consists of three runtime services: a REST API, a GPS device protocol listener, and a React web app. PostgreSQL with PostGIS is the database.

## Build & Run Commands

### Backend (.NET 10)

```bash
dotnet build                                      # Build entire solution
dotnet run --project backend/Navtrack.Api          # Start API (port 8080)
dotnet run --project backend/Navtrack.Listener     # Start Listener (ports 7001-7100)
```

### Frontend (React + Vite)

```bash
cd frontend && npm install                         # Install frontend deps
npm start                                          # Start web dev server (port 3000) from repo root
cd frontend/web && npm start                       # Same, from web directory
```

### Database

```bash
docker-compose -f docker-compose.dev.yml up        # Start PostgreSQL+PostGIS dev instance
```

Connection strings configured in `backend/Navtrack.Api/appsettings.Development.json` and `backend/Navtrack.Listener/appsettings.Development.json`.

### Tests

```bash
dotnet test                                        # Run all backend tests
dotnet test backend/Navtrack.Api.Tests             # API integration tests (uses Testcontainers)
dotnet test backend/Navtrack.Listener.Tests        # Listener protocol tests
dotnet test backend/Navtrack.Api.Services.Tests    # Service unit tests
dotnet test --filter "FullyQualifiedName~ClassName.MethodName"  # Run single test
```

Testing stack: xUnit, Testcontainers (PostgreSQL), Moq, Microsoft.AspNetCore.Mvc.Testing.

### API Client Generation

```bash
cd frontend/shared && npx orval                    # Regenerate API hooks from api.json
```

Config: `frontend/shared/orval.config.ts`. Input: `api.json` (OpenAPI spec at repo root). Output: `frontend/shared/src/api/`.

## Architecture

### Backend Structure

**Navtrack.Api** - ASP.NET Core Web API with controllers. Controllers delegate to typed `IRequestHandler` implementations. Uses NSwag for OpenAPI generation and IdentityServer4 for authentication.

**Navtrack.Api.Services** - Business logic layer using a Request/Response/Handler pattern (custom, not MediatR). Each handler has `Validate()`, `Handle()`, and optional `GetEvent()` methods. Base class: `BaseRequestHandler<TRequest, TResult>`.

**Navtrack.Api.Model** - DTOs/request/response models for the API.

**Navtrack.Listener** - Standalone hosted service that accepts TCP connections from 50+ GPS device protocols. Each protocol implements `IProtocol` (defines port, message boundaries) with a corresponding `IMessageHandler`. Pipeline: `ListenerHostedService` -> `ProtocolListener` -> `ProtocolConnectionHandler` -> `ProtocolMessageHandler`.

**Navtrack.Database** - Generic repository pattern over EF Core. `IGenericPostgresRepository<T>` provides CRUD operations.

**Navtrack.Database.Model** - EF Core DbContext and entity definitions. Uses PostgreSQL-specific features: JSONB columns, PostGIS geography types for coordinates, `gen_random_uuid()`.

**Navtrack.Database.Services** - Domain-specific repositories (e.g., `OrganizationRepository`, `AssetRepository`).

**Navtrack.Shared.Library** - Cross-cutting concerns. Key feature: `[Service]` attribute for automatic DI registration. `AddCustomServices<T>()` scans all assemblies starting with "Navtrack" and auto-registers services, eliminating manual DI configuration.

### Frontend Structure

Monorepo with npm workspaces: `frontend/web` (app) and `frontend/shared` (library).

**frontend/web** - Vite-bundled React app. Routing via react-router-dom v6 with `Paths.ts` constants. Layout patterns: TwoColumns, OneColumn, Settings. Components organized by domain: `/asset`, `/organizations`, `/teams`, `/users`, `/settings`, plus `/ui` design system.

**frontend/shared** - Reusable library containing:

- `api/` - Orval-generated React Query hooks (auto-generated from OpenAPI spec)
- `hooks/` - 84 custom hooks organized into `queries/`, `app/`, `util/`
- `state/` - Jotai atoms for global state (app config, asset UI state)
- `axios/` - Auth-intercepted axios instance with token management
- `translations/` - react-intl i18n JSON files

The model generated by Orval located in frontend/shared/src/api should not be edited, only generated by running "npm run generate" in the frontend/shared folder.

**Provider stack** (in order): ConfigProvider -> Jotai Provider -> QueryClientProvider -> AxiosConfigurator -> IntlProvider -> BrowserRouter.

**Key frontend patterns**: Jotai for state, TanStack React Query for server state, Formik + Yup for forms, Tailwind CSS v4 for styling, Leaflet + MapLibre for maps.
